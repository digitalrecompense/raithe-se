DEV-001: GBDT RANKER — HAND-TUNED TREES INSTEAD OF LAMBDAMART
================================================================================
Priority: CRITICAL
Location: crates/ranker/src/lib.rs
Issue Type: Architectural Deviation
Verification Status: ✅ CONFIRMED

Description:
The ranker uses hardcoded decision tree logic instead of loading a trained
LambdaMART model. The spec requires 300 LambdaMART trees for better ranking,
but the implementation uses only 10 hand-tuned heuristic trees.

Spec Requirement vs Current Implementation:
--------------------------------------------------------------------------------
| Aspect          | Spec Requirement         | Current Implementation     |
|-----------------|-------------------------|---------------------------|
| Algorithm       | LambdaMART (trained)     | Hand-tuned decision trees |
| Tree Count      | 300 trees (Section 7.2)| 10 trees                  |
| Max Depth       | 6 (Section 7.2)         | Not specified             |
| Learning Rate   | 0.05 (Section 7.2)      | 0.1                       |
--------------------------------------------------------------------------------

Code Evidence (ranker/src/lib.rs lines 252-381):
- 10 hand-tuned trees built in build_handtuned()
- Comment acknowledges: "These trees capture the key ranking signals until a
  LambdaMART-trained model is available"
- learning_rate: 0.1 (spec requires 0.05)

Impact: Ranking quality is severely degraded. The hand-tuned trees cannot match
the accuracy of a properly trained LambdaMART model. The ranking as is will
consistently underperform compared to specification.

Recommended Fix:
1. Train a LambdaMART model using a labeled relevance dataset
2. Export to ONNX format for inference
3. Implement model loading in RankingPipeline::step2_rerank()
4. Configuration must specify model path, not tree count/depth

DEV-002: NEURAL TOKENIZER — HASH-BASED PLACEHOLDER
================================================================================
Priority: CRITICAL
Location: crates/neural/src/lib.rs lines 54-160
Issue Type: Architectural Deviation
Verification Status: ✅ CONFIRMED

Description:
The SimpleTokenizer uses a hash function to generate token IDs deterministically.
Real ONNX transformer models expect specific vocabulary mappings from vocabulary
files (vocab.txt). The current implementation will not work with actual ONNX
models because token IDs will not match the model's expected vocabulary.

Code Evidence (neural/src/lib.rs lines 80-87):
```rust
let hash = word
    .to_lowercase()
    .bytes()
    .fold(5381u64, |h, b| h.wrapping_mul(33).wrapping_add(b as u64));
ids.push((hash % 29000 + 1000) as i64);
```

Code Comment Acknowledges (lines 44-53):
"// For production, this should load a real vocabulary file (vocab.txt) from
// the model directory. This implementation uses a hash-based approach..."

Spec Requirement vs Current Implementation:
--------------------------------------------------------------------------------
| Aspect          | Spec Requirement         | Current Implementation     |
|-----------------|-------------------------|---------------------------|
| Tokenizer       | Vocabulary-based (vocab.txt) | Hash-based placeholder |
| Token IDs       | Real vocabulary mapping | Deterministic hash        |
| Cross-Encoder   | MiniLM-L6-v2 compatible | Will not work             |
--------------------------------------------------------------------------------

Impact: Step 3 neural cross-encoder ranking CANNOT function. The neural pipeline
is completely non-functional until a real tokenizer is implemented.

Recommended Fix (CRITICAL Priority):
1. Write raithe originals or download real vocabulary files from Hugging Face for each model
2. Implement proper WordPiece/BPE tokenization
3. Load vocab.txt alongside .onnx model files
4. Test with actual ONNX model inference

DEV-006: PARSER — DOCUMENTATION VS IMPLEMENTATION MISMATCH

Issue Type: Documentation Inconsistency

The code comments state that "lol_html (streaming)" should be the primary parser

streaming approach of lol_html would be more memory-efficient. Additionally,

2. OR implement lol_html as primary with scraper fallback

DEV-007: INDEX WRITER HEAP SIZE TOO SMALL
================================================================================
Priority: MEDIUM
Location: crates/indexer/src/lib.rs line 162
Issue Type: Performance Configuration
Verification Status: ✅ CONFIRMED

Description:
The index writer heap is hardcoded to 50MB. According to Section 5.4, production
should use 1GB, with 50MB acceptable only for demo mode.

Code Evidence (indexer/src/lib.rs line 162):
"// Index writer with 50MB heap budget (Section 5.4)."
let writer = index.writer(50_000_000).map_err(...);

Spec Requirement vs Current Implementation:
--------------------------------------------------------------------------------
| Aspect          | Spec Requirement         | Current Implementation     |
|-----------------|-------------------------|---------------------------|
| Production Heap | 1GB (Section 5.4)       | 50MB                      |
| Demo Mode       | 50MB (acceptable)        | 50MB (default)           |
--------------------------------------------------------------------------------

Impact: Indexing throughput is significantly reduced. Larger heap allows more
documents to be held in memory before flushing to disk.

Recommended Fix:
1. Make heap size configurable via IndexerConfig
2. Default to 1GB for production deployments
3. Keep 50MB for demo/development
4. Implement auto-tuning based on available system memory

CORRECTED SEARCH PIPELINE ARCHITECTURE
================================================================================

The spec describes a three-step ranking pipeline. The CORRECT implementation is:

  User Query
      |
      v
  Query Understanding (Section 8)
  ├── Spell correction
  ├── Synonym expansion
  └── Intent classification

      |
      v
STEP 1 Retrieval (BM25F + ANN)
  ├── BM25F Search
  │   ├── Title: 5.0x
  │   ├── Body: 1.0x
  │   ├── Anchor: 4.0x
  │   ├── URL: 2.0x
  │   └── Headings: 2.5x
  ├── ANN Search (HNSW)
  │   ├── Bi-encoder embeddings
  │   ├── Cosine similarity
  │   └── Top-100 candidates
  └── RRF Fusion (k=60) → Top-1000 candidates

      |
      v
STEP 2 GBDT Reranking
  ├── Features:
  │   ├── BM25F score
  │   ├── PageRank (general + 4 topic-specific)
  │   ├── Document freshness
  │   ├── Query-term density
  │   ├── URL length
  │   ├── Click-through prediction
  │   └── 25+ additional features (Section 7.2)
  ├── LambdaMART Model: 300 trees, max_depth=6, lr=0.05
  └── Top-50 candidates for Step 3

      |
      v
STEP 3 Neural Cross-Encoder ⚠️ NOT INVOKED (DEV-005)
  ├── Model: MiniLM-L6-v2 (onnx)
  ├── Tokenize: Query + [SEP] + Document
  ├── Score: [CLS] token logits
  ├── Latency budget: 50ms, fallback at 100ms
  └── Top-20 reranked by cross-encoder score

      |
      v
  Final Ranking: Fuse GBDT Reranking + Neural Cross-Encoder scores

      |
      v
  Snippet Generation
  ├── Query term highlighting
  ├── IDF-weighted sentence selection
  └── Max 300 characters

      |
      v
  Results + Instant Answer

SPECIFICATION CORRECTIONS
================================================================================

The following corrections required are noted:

Correction 1: Index Writer Heap
Spec says: "1GB in production, 50MB in demo mode"
Correction: Make this configurable with sensible defaults per hardware tier.

Correction 2: Seed List Target
Spec says: "1000 friendly domains minimum"
Correction: This is correct but severely underspecified. The seed list must cover:
- Geographic diversity (not just Canada)
- Content type diversity (news, academic, commercial, government, reference)
- Quality tiers (high-authority vs. useful niche sites)

Correction 3: BM25F Configuration
Spec says: k1=1.2, b=0.75 (Section 7.1.1)
Correction: These are textbook defaults. Actual optimal values depend on corpus.
Make configurable.
